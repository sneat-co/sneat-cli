## AC: processing-is-swappable

**Verdict:** pass

**Justification:** The seam holds by compiler enforcement rather than discipline. `Processor` exposes `SendText`/`PressButton`, each returning `([]Reply, error)`; the concrete `processor` type is unexported and `NewProcessor` returns the interface, so a renderer is structurally unable to name an implementation. `internal/chattui` references only `chat.Processor` and `chat.Reply`; `chat.NewProcessor` appears exactly once, in the `RunChat` closure in `cmd/sneat/main.go`. Swapping in a server-backed processor touches that one line.

The strongest evidence is not an assertion but a demonstration: `chattui_test.go` already drives the entire renderer against its own `fakeProcessor`. A second implementation plugs in today, with zero renderer change — which is precisely the property this AC states.

`Reply.Keyboard` is genuinely `botkb.Keyboard` from bots-go-core v0.2.4 (confirmed via `go doc`), built as `[][]botkb.Button` rows. Callback data round-trips under `url.Parse` with dispatch on `URL.Path`. Reader failures come back as a wrapped error with nil replies, formatting no user-facing prose.
