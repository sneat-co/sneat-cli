## AC: interaction-is-unambiguous

**Verdict:** pass

**Justification:** All four risk areas were mutation-verified at HEAD rather than trusted.

**The `ctrl+c` carve-out is genuinely guarded.** `handleKey` matches `ctrl+c` *ahead* of `if m.pending { return m, nil }`. The precedent it mirrors — `internal/tui/confirm_screen.go`'s `if s.deleting { return s, nil }` — ignores every key including `ctrl+c`, so the departure is real. Copying that precedent verbatim (moving the pending lock ahead of `ctrl+c`) fails `TestCtrlCAlwaysQuits/while_a_reply_is_in_flight`. A user stuck on a hung backend with no way out is the failure this prevents.

**"Exactly one target holds focus" is fixed and asserted.** `syncInputCursor` blurs the input when the block holds focus; removing the blur (two focus indicators) fails `TestInputCursorFollowsFocus`.

**`esc` is load-bearing in both directions.** Making it quit from the block fails one test; making it not quit from the input fails another. Both-quit and neither-quit each fail.

**Every key has one defined meaning.** `handleInputKey`/`handleButtonKey` are separate per-focus tables matching `req:focus-and-keys` entry for entry; `clampCol` keeps the cursor on a button across ragged rows. Mutations to `clampCol`, the pending lock, `right`-wrap, `left` no-op, and `up`-at-row-0 were all killed.

9 of 10 mutants killed. The lone survivor (an unhandled key in the block falling through to the input) is a genuine **equivalent mutant** — the property is double-guarded by both the fallthrough and the blur — and was proven non-vacuous by removing both guards, which yields `input = "hellox"` and fails the test. The originating commit's "double-guarded and proven non-vacuous" claim was independently confirmed as honest.
